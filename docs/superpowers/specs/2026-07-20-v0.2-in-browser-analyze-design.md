# v0.2 design — In-browser "Analyze your own clip"

**Date:** 2026-07-20 · **Status:** approved design, pre-implementation
**Supersedes:** the Phase 6 deferral note in `SPEC.md` ("analyze your own clip in-browser is explicitly deferred to v0.2") — this is that v0.2.

## Context & goal

v0.1 (Jump Lab) is complete: CLI pipeline (video → MediaPipe pose → parquet →
metrics engine → outputs), validated physics, CI, Docker, and a live GitHub
Pages demo showing one precomputed real jump. v0.2 makes the demo
**interactive**: a visitor drops their own side-view clip onto the page and
gets the full Jump Lab analysis — pose, event detection, all metrics — computed
**entirely in their browser**. No upload, no server, no account.

Decisions taken during design (with the user):

| Decision | Choice | Why |
|---|---|---|
| Direction | In-browser analyze (over Film Room / Shot Charts / Jump Lab depth) | SPEC's own named next step; converts the static demo into an interactive product; best recruiter impact per unit effort |
| Scope | **Lean analyze** — upload → results, nothing persisted | Ships fastest; recruiting timeline; YAGNI |
| Devices | **Desktop-first**; phones get a functional but unoptimized experience | Recruiters are on laptops; mobile perf work deferred |
| Architecture | **A: fully client-side, TS engine port + golden-vector parity** (over Pyodide, over FastAPI backend) | Video never leaves the device (extends local-first identity to the web); free static hosting, no cold starts; ~10 MB payload vs 50–80 MB Pyodide; parity harness is itself a strong engineering artifact |

## Non-goals (v0.2)

- No backend, no accounts, no analytics, no persistence of user data (not even
  localStorage; the browser Cache API is used only for static assets — the pose
  model — never for user clips or results).
- No Pyodide; no physics in Python-in-WASM.
- No webcam/live mode.
- No mobile performance work beyond "it runs."
- No session history / progress tracking (that's the "mini-product" scope, later).
- No browser e2e suite (deliberate; see Testing).
- The Python CLI pipeline is untouched — no behavior changes, tests stay green.

## Architecture

```
file drop → <video> (object URL; browser applies rotation metadata natively)
  → muted playback with requestVideoFrameCallback (rVFC)
      → per presented frame: PoseLandmarker.detectForVideo(video, mediaTime_ms)
  → keypoint rows { t_ms, 33 × (x, y, visibility) }     # same shape as parquet rows
  → TS JumpSeries (Savitzky–Golay smoothing)
  → detectJumps → per-jump metrics (+ session aggregates)
  → results UI; entering height/reach recomputes calibrated metrics instantly
    (pure functions over kept keypoints — no re-pose)
```

- **Timestamps** come from rVFC `mediaTime` — real presentation timestamps.
  The v0.1 rule "use real timestamps, never assume fps" carries over verbatim.
- **Frame drops**: detected from gaps in the `mediaTime` deltas (and rVFC
  `presentedFrames` discontinuities); surfaced as a visible warning
  ("some frames were skipped — numbers may be less precise"), never silent.
- **Pose model**: `pose_landmarker_full` (~9 MB `.task`), fetched at runtime
  from Google's model hosting — the same runtime-download pattern the Python
  CLI already uses — and cached in the browser Cache API for repeat visits.
  The WASM runtime comes from the npm-bundled `@mediapipe/tasks-vision` assets
  served from our own origin (no third-party CDN for code; the model file is
  the one runtime fetch).
- **Known, accepted divergence**: browser MediaPipe and Python MediaPipe may
  produce slightly different keypoints for the same video (different decode +
  preprocessing paths). Parity is therefore enforced at the **engine boundary**
  (keypoints in → events/metrics out), not at the pose boundary. Documented in
  the page's limitations note.

## Repo layout & stack

```
web/                        # new top-level Vite + TypeScript app
  package.json
  vite.config.ts
  index.html                # single page; carries over demo/ design language
  public/
    sample/                 # precomputed sample-jump assets (moved from demo/assets/)
  src/
    engine/                 # 1:1 port of the Python engine (reviewable side-by-side)
      series.ts             #   ← jump/series.py  (JumpSeries + Savitzky–Golay)
      events.ts             #   ← jump/events.py  (detect_jumps, EventParams)
      metrics.ts            #   ← jump/metrics.py (pure metric functions)
      analysis.ts           #   ← jump/analysis.py (orchestration → per-jump results)
      __tests__/
        golden/             # committed JSON vectors exported by Python (see Parity)
        *.test.ts
    pose/
      extractor.ts          # rVFC loop + PoseLandmarker → keypoint rows
    ui/
      app.ts, overlay.ts, cards.ts, errors.ts, progress.ts
    style.css               # carried over / extended from demo/ CSS
tools/
  export_golden_vectors.py  # Python: fixtures + real sample keypoints → golden JSON
```

- **No framework** (no React): the existing demo page is hand-rolled HTML/CSS/JS
  with an established brand; v0.2 continues it. TypeScript `strict`, ES modules.
- **Tooling**: `vitest` (unit + parity tests), `tsc --noEmit`, `prettier --check`.
  Mirrors the Python side's ruff/mypy/pytest discipline without new ceremony.
- **`demo/` retirement**: `web/` replaces `demo/` as the deployed page.
  `demo/build_demo.py`'s artifact-generation role moves to
  `tools/refresh_sample_assets.py` (refreshes `web/public/sample/` from a
  completed run); its HTML-templating role is deleted. The old `demo/` folder
  is removed once `web/` deploys.
- **Committed-video exception moves**: the deliberate narrow exception that
  allows committing the *rendered* sample overlay video moves from `demo/` to
  `web/public/sample/` — both `.gitignore` and the
  `.claude/hooks/block_video_commits.sh` allowlist must be updated in the same
  commit that moves the assets, or the hook will block the move.

## Engine port (what exactly is ported)

| Python | TS | Notes |
|---|---|---|
| `jump/series.py` | `engine/series.ts` | JumpSeries construction from keypoint rows; Savitzky–Golay implemented from scratch — **must match scipy's `savgol_filter` defaults including `mode='interp'` edge behavior** (polynomial fit at the edges), since event detection near clip boundaries depends on it |
| `jump/events.py` | `engine/events.ts` | `EventParams`, `JumpEvent`, `detect_jumps` incl. the v0.1 review fixes (boundary guard, planted-foot baseline) |
| `jump/metrics.py` | `engine/metrics.ts` | All pure metric functions + calibration constants (`EYE_HEIGHT_RATIO` etc.), touch height included |
| `jump/analysis.py` | `engine/analysis.ts` | Orchestration: series → jumps → per-jump metric dicts + session aggregates, incl. graceful no-countermovement handling |

Not ported: `pose/extractor.py` (replaced by `@mediapipe/tasks-vision` in
`pose/extractor.ts`), `viz/overlay.py` (replaced by live canvas overlay),
`jump/report.py` (charts/JSON files are CLI outputs; the page renders cards).

Port discipline: pure functions, no I/O in `engine/`; every metric function's
doc comment states units; every formula keeps its derivation comment; image-y-
grows-downward sign notes carried over.

## Parity harness (the correctness guarantee)

1. `tools/export_golden_vectors.py` (Python, runs under `uv`):
   - runs the **existing synthetic fixtures** (parameterized jumps with known
     ground truth, plus edge cases: no jump, two jumps, missing frames) through
     the Python engine, and
   - runs the **real sample clip's** saved keypoints through the Python engine,
   - writes JSON vectors: `{ input: keypoint rows + params, expected: events
     (frame indices + times) + metrics (values or null) }` into
     `web/src/engine/__tests__/golden/`.
2. Vitest parity suite: for every vector, run the TS engine on `input`, assert:
   - event **frame indices exactly equal** (both sides are IEEE-754 float64
     running the same algorithm; fixtures are designed away from threshold
     knife-edges),
   - metric values within **1e-6 relative tolerance, with a 1e-9 absolute
     floor** for near-zero values (e.g. arm-swing timing can legitimately be
     ~0); null must match null.
3. CI drift guard: the Python job re-runs the exporter and fails if the
   regenerated vectors differ from the committed ones (`git diff --exit-code`).
   So: Python engine change → vectors change → committed vectors stale → CI red
   until vectors are re-exported **and** the TS suite passes against them.
   Drift in either implementation is caught on every push.

## Page UX (single page, current branding)

"Current branding" is the **court-graphic identity** shipped to `demo/` on
2026-07-20: court orange `#D14A24` / ink `#16232B` / paper `#FBFBF8` / teal
`#0E6672` (+ antenna red `#C8102E` in small doses); Barlow Condensed for
display and stat numerals, Barlow for body, Spline Sans Mono for data; the
red/white antenna tick as section marker; one full-bleed court-orange panel as
the single bold moment; box-score-style metric tables; hairline structure —
no cards, pills, or shadows. The v0.2 app carries this identity forward.

Order top to bottom:

1. **Hero + sample analysis** — the current precomputed real-jump demo,
   unchanged in spirit: instant, zero-effort path for visitors with no clip.
2. **"Analyze your own clip"** — drop zone (+ file picker) with the capture
   protocol tips inline (side view, tripod, whole body, one athlete, 60 fps).
   Privacy line displayed prominently: *"Analysis runs entirely on this device —
   your video is never uploaded."*
3. **Progress** — determinate bar while pose runs (frames processed, detected
   fps, elapsed); cancel button.
4. **Results** — the uploaded video replaying with a live canvas skeleton
   overlay and LOAD/TAKEOFF/LANDING flags synced to playback; metric cards per
   jump (height, touch height when reach given, depth, loading time, approach
   velocity, arm-swing timing); session summary line when multiple jumps;
   optional height-cm / reach-cm inputs that recompute calibrated metrics
   instantly. An "analyzed on-device" badge. A compact limitations note
   (monocular, side-view, equal-level takeoff/landing) linking to VALIDATION.md.

Desktop-first: layout and perf tuned for laptop; on phones the page functions
but shows a "best experienced on desktop" hint near the drop zone.

## Error handling (user-facing, specific)

| Failure | Behavior |
|---|---|
| Unsupported codec (e.g. HEVC `.mov` on some browsers) | Detect via `<video>` error/no-frames; message + "export as MP4 (H.264)" tip |
| No person detected / chronically low visibility | Message + capture-protocol tips |
| No jump detected | Message ("we tracked the athlete but found no jump") + tips |
| Clip too long | Soft cap ~60 s: warn + offer to analyze the first 60 s |
| Frame drops during processing | Visible warning banner on results; numbers still shown |
| WASM/GPU unavailable | Graceful message; page's sample section still works |
| Calibration fails (no clean standing pose) | Calibrated metrics shown as "—", uncalibrated ones still computed (same graceful degradation as the Python engine) |
| Jump with no countermovement (block jump) | Depth/loading "n/a", everything else computed (inherited fix) |

## Testing

1. **Golden-vector parity suite** (the core): TS engine must reproduce the
   Python engine's outputs on synthetic + real vectors — see Parity harness.
2. **TS unit tests** for browser-agnostic glue: Savitzky–Golay against known
   coefficient tables and against scipy-generated cases (incl. edge mode);
   frame-collector gap handling and timestamp monotonicity; calibration math.
3. **No browser e2e in v0.2** (deliberate, lean): the pre-deploy manual check
   is the sample clip through the real page. Playwright smoke is a noted
   candidate for v0.3.
4. Python suite untouched and green; CI adds a Node job (tsc, prettier check,
   vitest) and the vector drift guard to the Python job.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Savitzky–Golay edge behavior mismatch vs scipy | Implement `mode='interp'` explicitly; unit-test against scipy-generated cases including edges; parity vectors exercise clip-boundary jumps |
| Browser vs Python keypoint divergence confuses "parity" | Parity defined at engine boundary only; page limitations note says browser numbers may differ slightly from CLI on the same clip |
| rVFC frame drops on slow machines corrupt timing | Drop detection + visible warning; muted 1× playback (inference throttles decode naturally); WebCodecs decode is the noted v0.3 upgrade if this bites |
| HEVC iPhone footage fails on some desktop browsers | Explicit codec error path + conversion tip; sample path always works |
| Model hosting (Google) unavailable | Clear error + retry; Cache API keeps repeat visits working; same dependency the CLI already has |
| Scope creep toward mini-product | Non-goals section above is binding for v0.2 |

## Done when (success criteria)

1. A visitor on desktop Chrome/Safari/Firefox drops a clean side-view clip and
   gets events + all metrics with the skeleton overlay replay, fully offline
   after initial asset load — verified with the real sample clip.
2. Vitest parity suite green against freshly exported vectors; CI (both jobs)
   green; Pages deploys `web/dist` and the live URL serves the new page with
   the sample path intact.
3. Entering height/reach updates calibrated metrics without re-running pose.
4. Every error path in the table above reachable and showing its specific
   message (manually exercised).
5. SPEC rev 3, CLAUDE.md, README, DEVLOG updated.
